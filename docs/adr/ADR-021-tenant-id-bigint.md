# ADR-021 — `tenants.id` como BIGINT, `zitadel_org_id` mantido à parte

**Status:** Aceito
**Data:** 2026-05-24
**Decisores:** Allysson Pereira

## Contexto

A versão inicial do schema usava `tenants.id UUID PRIMARY KEY` (e `tenant_id UUID NOT NULL` em todas as tabelas filhas: `users`, `pessoas`, `pipelines`, `etapas`, `oportunidades`, `invites`, `setores`, `funcoes_pessoas`, `tipos_empresa`, `tipos_projeto`, `unidades`, `empreendimentos`, `ofertas`, `tipos_pagamento`).

Argumentos pró-UUID: distribuído (gerar id no client), opaco em URLs, fácil colar entre bancos.

Argumentos contra (para `tenants` especificamente):

1. **Cardinalidade baixa**: o sistema terá talvez 100–10k tenants (escritórios de arquitetura). UUID v4 são 16 bytes; um BIGINT são 8 bytes. Cada linha de cada tabela filha carrega 8 bytes a mais, multiplicado por **milhões** de oportunidades, pessoas, etc.
2. **Índices pesam menos**: índices em `tenant_id` aparecem em quase todas as tabelas. Reduzir o tamanho da chave estrangeira melhora cache hit ratio e performance de joins.
3. **Snowflake do Zitadel já é externo**: o `zitadel_org_id` (string ~18 dígitos) já é o identificador opaco que o IdP nos dá. Reaproveitar ele como PK significaria PK com formato variável e fora do controle do banco. Mantê-lo como coluna separada (UNIQUE) deixa o ID externo isolado da nossa PK.
4. **UX de logs**: id BIGINT pequeno (`tenant_id=1` no log) é mais legível e diagnosticável que `tenant_id=00000000-0000-0000-0000-000000000001`.
5. **URLs**: pra TKWS OS a URL nunca expõe `/tenants/{id}` para usuário final — multi-tenancy é via JWT (ADR-019), não path param. Não há ganho prático de opacidade UUID.

O custo da migração era um refactor moderado em ~40 arquivos Java + migrations + frontend schemas, feito de uma vez antes de a base de código crescer.

## Decisão

**`tenants.id` é `BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`. Todas as tabelas filhas usam `tenant_id BIGINT NOT NULL REFERENCES tenants(id)`. `zitadel_org_id VARCHAR(255) UNIQUE NOT NULL` continua como coluna separada em `tenants` para o mapping `IdP → tenant local`.**

Implicações:

- `TenantId` é `record TenantId(long value)`. Validação `value > 0`.
- `Tenant` agregado é **transiente** até o adapter persistir: `Tenant.create(...)` retorna `id == null`; o adapter chama `tenant.assignIdIfTransient(TenantId.of(saved.id))` após o INSERT, que atribui o id real e dispara `TenantCreatedEvent`.
- `TenantContext(long tenantId, String zitadelOrgId)` — o `zitadelOrgId` continua para que o backend possa logar/auditar a origem do tenant context.
- Outros agregados (`Pessoa`, `Pipeline`, `Etapa`, `Oportunidade`, `Invite`, etc.) usam `long tenantId` direto (não há value object — é um identificador primitivo do dono do dado).
- Header de override `X-Tenant-Id` aceita apenas BIGINT positivo (era UUID antes).
- `feature_flags.enabled_for_tenants` / `disabled_for_tenants` viram `BIGINT[]` (eram `UUID[]`).
- Frontend: `tenantId: z.number().int().positive()` nos schemas Zod (era `z.string().uuid()`).

PKs de **outras** entidades (Pessoa, Oportunidade, etc.) continuam UUID. A escolha de BIGINT é específica para `tenants` porque ele é uma dimension table de baixa cardinalidade referenciada em todas as outras tabelas.

## Alternativas consideradas

1. **Reutilizar `zitadel_org_id` como PK (snowflake string)**. Rejeitado: amarra a PK ao formato externo do IdP. Se um dia trocarmos de IdP, precisamos refazer todo o referencing. Migração de Zitadel também não preserva ids (instâncias novas geram novos snowflakes).

2. **Manter UUID em tenants.id**. Rejeitado: custo de armazenamento/índice multiplicado em todas as tabelas, sem ganho prático já que o IdP-id externo (`zitadel_org_id`) já existe como coluna separada.

3. **BIGINT local + zitadel_org_id à parte** (esta ADR) ✓. Aceito: chave interna eficiente, identificador externo preservado, mapping `zitadel_org_id → tenant_id` cacheado em memória pelo `TenantIdResolverAdapter`.

## Consequências

### Positivas

- **Storage**: 50% menos bytes por linha em colunas `tenant_id`. Em uma tabela de 10M oportunidades isso são ~80MB de redução só na coluna + índice.
- **Performance**: índices BIGINT cachem melhor (mais entries por página).
- **Legibilidade**: `tenant_id=1` em logs/diagnóstico.
- **Independência do IdP**: trocar de Zitadel para outro IdP no futuro é uma operação de update no `zitadel_org_id`, sem tocar nas FKs.

### Negativas / Trade-offs

- **Ids previsíveis**: BIGINT IDENTITY é sequencial. Em ambientes onde a contagem de tenants é informação sensível (ex.: vazar quantos clientes a TKWS tem), o id revela isso. Mitigação: o tenant_id **nunca** aparece em URL pública ou body de API para usuário final — só via JWT/header (ADR-019). Para sistemas admin que listam tenants, isso é informação interna.
- **Agregado transiente**: `Tenant.create()` retorna `id == null`, exigindo `assignIdIfTransient` pelo adapter. Mais complexo que UUID gerado no construtor. Mitigação: encapsulado, testado (`TenantTest$WhenCreating.assignIdIdempotente`).
- **Refactor amplo**: ~40 arquivos Java + 5 migrations + 13 schemas Zod do frontend. Custo pago em PR única antes da base crescer.

### Riscos

- Se eventualmente reintroduzirmos PKs distribuídas (ex.: sharding por região), precisaremos repensar — BIGINT IDENTITY não é shard-friendly. Hoje é trade-off aceitável; revisitar quando >10 regiões ou >1M tenants.

## Aplicação retroativa

Migrations V1, V2, V4, V5, V6 e R__seed_dev foram **reescritas in-place** (não migration nova) porque o projeto está em fase inicial — não há banco produtivo ainda. Em produção, esse tipo de mudança exigiria `V7__migrate_tenant_id_to_bigint.sql` com janela de manutenção (rebuild de todos os FKs + índices). Documentar isso aqui para a IA não recriar esse padrão de "reescrever V1" depois que a app for pra produção.
